# TASK-10 · 高斯群组共创内容图片转录

**目标**：把一份由 256 张 PNG 图片组成的印象笔记导出 docx，转录为 Opus 可低成本快速阅读的结构化 markdown。

**为什么交给视觉模型**：256 张高清截图直接喂给 Opus 成本约 50万 tokens；让视觉模型先转录为文本，Opus 仅消费几千 tokens 就可以读完全部内容并做提炼/决策。**这是典型的"用便宜模型预处理 + 用贵模型决策"的成本优化模式。**

---

## A. 输入说明

源数据位于：

```
knowledge-base/_inbox/unzipped/word/media/
├── image1.png
├── image2.png
├── ...
└── image256.png
```

文件按数字编号顺序对应原 docx 的页面顺序。原 docx 是中国 3DGS 中文社群成员共创的印象笔记笔记本导出文件，含**结构化教程目录、采集 / 训练 / 发布教程文字稿、群友共创实战分享、资源链接汇总**等。

**绝大多数图片是中文内容**——印象笔记或聊天截图风格的页面，含正文、流程图、代码片段、网址、图标等。

---

## B. 输出位置

按批输出到：

```
knowledge-base/_inbox/transcribed/batch-NN.md
```

每批处理 20 张图（共约 13 批）。一批一个 markdown 文件。

例：`batch-01.md` 装 image1 ~ image20，`batch-02.md` 装 image21 ~ image40，依此类推。最后一批 `batch-13.md` 装 image241 ~ image256。

---

## C. 输出格式（每张图一个 section，固定结构）

```markdown
## image_NNN（NNN 是 3 位补零编号）

- **type**: 印象笔记文章 | 教程文字稿 | 工具截图 | 聊天对话 | 资源链接列表 | 流程图 | 代码片段 | 数据集示例 | 其他
- **section_title**: <图中可见的章节标题，如「第一部分：采集」>
- **contains_pii**: true | false（是否含微信昵称 / 个人网址 / 私域群名等个人识别信息）
- **handling_notes**: <如做了去人格化处理，简述替换规则>

### 转录正文

<图中所有可见文字，verbatim 中文。保留原段落结构、序号、缩进。>

<注意：流程图 / 截图中的小字也要尽力转录；明显是 UI 装饰元素（如导航栏「印象笔记」「保存到 APP 查看」等）可省略，但 hero 文案、按钮上的关键文字必须转录。>

### 外部链接（如有）

- 公开资源（GitHub / 官网 / 公开博客 / 公开平台）：
  - <verbatim URL>
- 私域内容（百度网盘 / 微信群文件 / 个人未公开网站）：
  - [已隐去：私域分享]（保留链接但不公开化）

### 图片中的代码 / 命令 / 文件清单（如有）

```bash
<代码 verbatim>
```

### 图中提及的工具 / 产品名（去重列表）

- <工具1>
- <工具2>
```

---

## D. 去人格化硬规则（重点）

这份内容的来源——3DGS 中文社群——是私域共创成果。我们要在网站里使用提炼后的"知识"，但**绝对不要暴露社群成员个人身份**。

### 必须去人格化的内容

| 类型 | 处理方式 | 例子 |
|---|---|---|
| **微信昵称 / @某某** | 替换为 `[群友1]`、`[群友2]`（同一图内同一人保持一致） | `@koko 老师` → `[群友1]` |
| **个人微信号 / QQ 号** | 全部隐去 | 写为 `[已隐去：私人联系方式]` |
| **群名 / 群号** | 替换为 `[3DGS 中文社群]` | "高斯泼溅交流群" → `[3DGS 中文社群]` |
| **个人未公开网站** | 标注但隐去 URL | `www.v4ko.cn` → `[群友个人网站，已隐去]` |
| **聊天截图中的对方头像 / 头像旁的姓名** | 隐去 | 描述图片时用 `[群友 A 发言]` |
| **私人通讯录截图** | 全部隐去具体内容 | 仅描述「该图为某成员通讯录截图」 |

### 可以保留的内容

| 类型 | 保留原因 | 例子 |
|---|---|---|
| **GitHub 用户名 + 公开仓库** | GitHub 公开信息，已与开源项目绑定 | `github.com/fenghuayumo/DIVSHOT` 全保留 |
| **公开产品 / 公司名** | 商业实体公开信息 | "点映科技"、"知天下"、"BSD Studio" 等 |
| **公开论文作者署名** | 学术规范，公开信息 | 论文 PDF 中的作者列表 |
| **印象笔记文章标题** | 标题本身没有 PII | "3D 高斯：从拍摄到生成的全流程" |
| **教程文字稿正文** | 知识本身去人格化 | 拍摄技巧、训练方法、代码片段 |

### 灰色地带（标注存疑，让 Opus 决定）

- 群友提到的"小品牌"或"个人工作室"：保留品牌名但不写明跟某群友的关联
- 群友推荐的"百度网盘共享文件"：保留链接但标注 `[私域分享，待 Opus 决定是否引用]`
- 群友的实战案例图（拍摄的物品 / 场景）：保留描述但去掉拍摄者归属

---

## E. 反幻觉硬规则

1. **只写图中可见的内容**：不要"补全"图中没说但你"猜得到"的内容
2. **图中文字模糊不清**：标注 `[模糊不可辨认]`，不要瞎猜
3. **图中含截图嵌套**（如印象笔记页面里又截了一个软件界面）：分别转录，标注「内层截图：xxxx」
4. **代码块严格 verbatim**：包括反引号、分号、缩进、引号——一字不改
5. **URL 严格 verbatim**：不要"修复"看似有错的 URL（如 `https://github.com/xxxx`），原样转录
6. **如某图全是装饰性图片**（如品牌 banner、纯艺术图）：type 标 `其他`，转录正文写「该图为装饰性图片，无可转录文字内容」

---

## F. 转录时的特别注意事项

### 流程图与思维导图

很多印象笔记页面包含分支型流程图。请按从左到右、从上到下的层级关系转录，使用 markdown 嵌套列表表示层级。例：

```markdown
- 采集（核心入口）
  - 激光辅助采集
    - K1 设备
    - P1 设备
  - 纯视觉采集
    - 单反相机
    - 手机
    - 全景相机
```

### 印象笔记的"智能摘要"

印象笔记原始页面顶部常有绿色背景的"智能摘要"框。请把这部分转录到 `### 智能摘要` 子标题下，与正文分开。

### 含"立即开始"、"免费试用"、"登录" 等按钮

这是 UI 元素而非内容，可在 `转录正文` 末尾加 `### UI 元素` 节标注，列出按钮文字即可，不必占大段空间。

### 工具 logo 或 banner 图

转录工具名 + 标语 + 显示的功能徽章（如 `Discord` / `License GPLv3` / `CUDA 12.8+` 等），不必描述视觉风格。

## H. 任务执行节奏

总共 256 张 → 13 批 → 每批 20 张（最后一批 16 张）。

按 batch 顺序执行：batch-01.md → batch-02.md → ... → batch-13.md。

**每完成 3 批回复一次**给 Opus（不必每批通知），让 Opus 抽样校对，避免一路跑到最后才发现转录质量问题需要返工。

---

## I. 给 Gemini Pro 用户的特别说明（如本任务由 Gemini 执行）

如果用户使用 Gemini Pro 多模态网页版执行本任务：
- 一次性上传 20 张图（一批），并把本文件作为系统指令
- Gemini 输出后，用户复制 markdown 内容到 `_inbox/transcribed/batch-NN.md`
- 自检章节也复制进去

如果使用 AUTO（Cursor 内置）执行本任务：
- AUTO 直接读取 `_inbox/unzipped/word/media/imageNN.png`
- AUTO 直接写入 `_inbox/transcribed/batch-NN.md`
- 完成 3 批后 stop 等待 Opus 确认

## J. 完成后

13 批全部完成后，回复一份**总结报告**：

1. 整份文档主要分为几个章节（如「采集篇 / 训练篇 / 发布篇 / 群友共创 / 资源汇总」）
2. 每个章节占的图片范围（如「采集篇 image_001 ~ image_065」）
3. 整份文档提及的全部工具去重列表（如 50 个 unique tool names）
4. 整份文档提及的全部 GitHub 仓库（公开资源）去重列表
5. 含 PII 的图片总数（占 256 的百分比）
6. 你认为对 INKTOYS 网站建设最具价值的 5-10 个章节 / 图片范围

完成后通知 Opus 接手做"提炼 → 结构化入库"工作。
